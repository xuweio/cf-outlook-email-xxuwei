# 本地正常，部署后登录接口 500

- **日期**：2026-06-08
- **类型**：使用问题 / 部署配置缺失（非代码逻辑 bug），附文档改进
- **影响**：用户本地运行正常，部署到生产后，`POST /api/auth/login` 返回 500，登录页提示「网络错误，请重试」。

## 现象

- 登录页输入密码 → 接口 `…workers.dev/api/auth/login` 返回 **500 Internal Server Error**。
- 前端因拿不到正常 JSON，显示「网络错误，请重试」。
- 本地 `pnpm run dev` 一切正常。

## 根本原因

登录流程第一步 `verifyPassword()` 就要查数据库的 `settings` 表，且随后用
`COOKIE_SECRET` 签发会话 Cookie。生产环境若少配，会直接抛异常。又因为
`src/index.ts` **没有全局错误处理（`app.onError`）**，任何异常都变成裸 500，
信息不可读。

本地能跑、生产 500 的差异通常来自以下三项之一（按概率）：

1. **远程 D1 未迁移**：只跑了本地 `db:migrate:local`，漏了
   `wrangler d1 migrations apply outlook-email-db --remote`。远程库无
   `settings` / `accounts` 表 → 查表抛错 → 500。
2. **密钥未设**：未对生产执行 `wrangler secret put ADMIN_PASSWORD` /
   `COOKIE_SECRET`（本地 `.dev.vars` 不会上传）。缺 `COOKIE_SECRET` 在签发
   Cookie 时抛错 → 500。
3. **`wrangler.toml` 的 `database_id` 填错或仍是占位符** → D1 绑定无效 → 查询抛错 → 500。

## 解决（用户侧）

```bash
pnpm exec wrangler d1 migrations apply outlook-email-db --remote
pnpm exec wrangler secret put ADMIN_PASSWORD
pnpm exec wrangler secret put COOKIE_SECRET
```

确认 `wrangler.toml` 的 `database_id` 是自己 `d1 create` 出来的真实 ID。
以上均作用于线上资源，无需启动本地 dev；secret 改动下次请求即生效。

### 坑：`npm exec ... --remote` 仍跑本地

有用户用 `npm exec wrangler d1 migrations apply outlook-email-db --remote`，结果输出
`Resource location: local` + `No migrations to apply!`——`--remote` 被 `npm exec` 当成
自己的参数吃掉了，没传给 wrangler。这是 `npm exec` 的参数处理行为（与终端无关，
bash/cmd/PowerShell 都一样），`npx` / `pnpm exec` 不受影响。解决：

```bash
npx wrangler d1 migrations apply outlook-email-db --remote
# 或
npm exec -- wrangler d1 migrations apply outlook-email-db --remote
# 或（本项目默认）
pnpm exec wrangler d1 migrations apply outlook-email-db --remote
```

成功时不再出现 `Resource location: local`。另注意别从文档复制 `--`，可能被自动转成长破折号 `—`。

## 看真实报错的坑（国内网络）

`pnpm exec wrangler tail` 会报 `connect ETIMEDOUT ...:443`：tail 会话已创建，
但日志流长连接被墙连不通。绕过方式：

- 挂代理：`HTTPS_PROXY=http://127.0.0.1:端口 pnpm exec wrangler tail`
- 或用网页版：Cloudflare 控制台 → Workers & Pages → worker → **Logs** →
  **Begin log stream**（浏览器直连，国内可用）。

## 代码加固（已完成，分支 `feat/error-handling`）

- `src/index.ts` 增加 `app.onError`：把未捕获异常统一转成 JSON 500，并识别常见误配——
  「no such table / D1_ERROR」→ 提示远程库未迁移；「crypto / HMAC / key」→ 提示密钥未设；
  其余给通用「请检查部署配置」。因此登录页（`await res.json()`）能直接显示真实原因，
  不再因解析 HTML 失败而显示「网络错误」。
- `src/routes/auth.ts` 登录前置检查 `COOKIE_SECRET`，缺失时直接返回明确提示。
- 本地实测四种情况均符合预期：正常登录成功、密码错 400、settings 表缺失 → `DB_NOT_READY`、
  缺 `COOKIE_SECRET` → `CONFIG_MISSING`。
